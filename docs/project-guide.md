# AI 패션 코디 추천 프로젝트 기획 가이드

> 이 문서는 Cursor AI가 프로젝트 맥락을 이해하고 코드를 작성할 때 참고하는 기획/설계 문서입니다.

## 1. 프로젝트 개요

**한 줄 요약**: 사용자의 쇼핑앱 찜/구매 데이터를 기반으로, 날씨와 TPO(Time·Place·Occasion) 조건에 맞는 코디를 AI가 추천해주는 서비스.

**문제 정의**: "옷은 많은데 오늘 뭐 입지?" — 이미 구매/찜한 옷들이 있는데 상황에 맞는 조합을 고르기 어려운 문제를 해결.

**프로젝트 성격**: 사이드 프로젝트 / 포트폴리오. 기존에 LLM·RAG 프로젝트(예: ResearchMind)를 진행해왔기 때문에, 이번 프로젝트는 **의도적으로 멀티모달/비전 임베딩 영역**을 메인 학습 목표로 잡는다. 텍스트 RAG 포트폴리오와 비전/멀티모달 포트폴리오가 세트로 짝을 이루는 것이 목표.

## 2. 핵심 기술 방향

**결론: LLM만으로 MVP는 가능하지만, 프로젝트의 학습 목표상 멀티모달(CLIP류 임베딩)을 메인으로 채택한다.**

- 쇼핑몰 상품은 이미 메타데이터가 풍부함 (카테고리/색상/소재/핏/스타일 태그 등 텍스트). 이것만으로도 LLM 기반 MVP는 동작 가능.
- 다만 **시각적 조화 판단**(색·패턴이 실제로 잘 어울리는지)은 메타데이터 라벨만으로는 놓치는 부분이 많음 → 여기서 CLIP류 멀티모달 임베딩이 진짜 값어치를 함.
- 옷장 사진을 직접 업로드하는 확장 기능을 고려할 경우, 메타데이터가 없는 이미지의 속성 추출에도 비전 모델이 필요.

### 모델 역할 구분 (중요 — 개념 정리)

| 구분 | 역할 | 사용 모델 예시 |
|---|---|---|
| **속성 추출 (VLM)** | 이미지 → 텍스트 속성(카테고리, 색상, 소재 등) 추출. 프롬프트로 태스크 지정 → JSON 구조화 출력 → 파싱/재시도 → 평가. 텍스트 LLM 다루던 파이프라인과 구조 동일 (입력만 이미지로 바뀜) | Qwen-VL, LLaVA, Claude Vision |
| **코디 조화 스코어링 (임베딩 모델)** | 옷 이미지를 벡터화하고, 조합 간 유사도/조화도를 계산. VLM(생성형)이 아니라 **비전-언어 임베딩 모델**임에 주의 | CLIP, FashionCLIP, KoCLIP |
| **최종 추천 설명 생성 (LLM/VLM)** | 임베딩 스코어링 결과를 받아 "왜 이 조합인지" 설명 텍스트 생성 | 기존 LLM 파이프라인 재사용 |

> CLIP과 VLM(생성형)은 다른 카테고리다. CLIP은 임베딩 모델, VLM은 이미지+텍스트를 받아 생성하는 모델. "코디 추천은 CLIP 임베딩이 중심이고, 설명 생성에 LLM/VLM을 붙이는 구조"로 이해할 것.

## 3. 전체 파이프라인 아키텍처

```
[사용자 찜/구매 데이터 or 옷장 이미지]
        ↓
① 메타데이터 필터링 (날씨/TPO 조건으로 후보 좁히기)
        ↓
② 멀티모달 임베딩으로 조합 조화 스코어링 (CLIP/FashionCLIP)
        ↓
③ LLM으로 최종 추천 + 추천 이유 설명 생성
        ↓
(선택) VLM으로 이미지 기반 속성 보강/설명 강화
```

각 단계에서 가장 잘하는 모델에게 역할을 위임하는 구조.

## 4. 기술 스택

- **벡터 DB**: Qdrant (기존 RAG 프로젝트에서 사용 경험 재활용, 여기서는 텍스트 대신 이미지 임베딩 저장)
- **오케스트레이션**: LangGraph — "조건 파싱 → 후보 필터 → 조화 스코어링 → 코디 생성/설명" 파이프라인 구성
- **임베딩 모델**: CLIP / FashionCLIP / KoCLIP (한국 패션 도메인 특화 고려 시)
- **LLM**: 기존 사용 LLM 재사용 (추천 이유 설명 생성)
- **VLM (선택)**: Qwen-VL, LLaVA, Claude Vision 등 — 옷장 사진 속성 추출용

## 5. 데이터 전략

### 문제
지그재그/에이블리 등은 공개 API가 없고, 개인 구매내역 크롤링/스크래핑은 약관 위반 + 개인정보 이슈 발생 가능.

### 선택지
1. **사용자가 직접 옷장을 등록**하는 방식 (이미지 업로드)
2. 제휴/오픈뱅킹식 데이터 연동 (실서비스화 시 고려, 현재는 우선순위 낮음)
3. **PoC/포트폴리오용 공개 데이터셋 + 목업 데이터**로 파이프라인 증명 (현재 채택 방향)

### 추천 공개 데이터셋
- **Polyvore** — "어울리는 코디 조합 세트"가 라벨링되어 있어 조화도 학습/평가에 바로 활용 가능. **우선 채택 추천.**
- **DeepFashion** — 패션 이미지 속성/카테고리 데이터
- **K-Fashion** (AI Hub) — 한국 패션 특화 데이터셋

## 6. 개발 우선순위 (학습 목표 기준)

1. **메인 (새로 배우는 축)**: CLIP류 멀티모달 임베딩으로 옷 이미지 벡터화 + 조합 조화도 스코어링. 프로젝트에서 가장 낯선 영역이므로 여기부터 먼저 완성.
2. **재활용 축**: Qdrant 벡터 DB, LangGraph 파이프라인 — 기존 RAG 프로젝트 경험 그대로 적용 가능 (모달리티만 텍스트→이미지로 전환).
3. **확장 축 (여유 있으면)**: VLM 기반 속성 추출을 앞단에 붙여 이미지만으로도 메타데이터 없이 코디 추천 가능하게 확장. 이 부분은 기존 프롬프트 엔지니어링/구조화 출력(JSON) 경험과 파이프라인 구조가 겹침 — 완전히 새로운 것이 아니라 입력 모달리티만 바뀐 것으로 이해.

> 개발 순서 권장: **CLIP 조화도 스코어링 파이프라인 완성 → 이후 VLM 속성 추출을 앞단에 붙이는 순서**로 진행. (CLIP 쪽이 안 해본 영역이라 우선순위 높음)

## 7. MVP 범위 제안

- [ ] Polyvore 데이터셋으로 코디 조합 데이터 확보
- [ ] CLIP/FashionCLIP으로 아이템 이미지 임베딩 후 Qdrant에 저장
- [ ] 날씨/TPO 조건 입력 → 메타데이터 필터링 로직
- [ ] 필터링된 후보군 내에서 임베딩 유사도 기반 조화도 스코어링
- [ ] LLM으로 최종 Top-N 추천 + 추천 이유 텍스트 생성
- [ ] (확장) 사용자 이미지 업로드 → VLM 속성 추출 → 파이프라인 앞단 연동

## 8. 주의할 개념 정리 (헷갈리기 쉬운 부분)

- **CLIP ≠ 생성형 VLM**: CLIP은 비전-언어 "임베딩" 모델이고, LLaVA/Qwen-VL 같은 생성형 VLM과는 카테고리가 다름. 코드/문서에서 혼용하지 말 것.
- **"겹친다"의 의미**: VLM 속성 추출이 기존 LLM 작업과 "모델"이 겹치는 게 아니라, 그것을 감싸는 파이프라인 구조(프롬프트 지정 → JSON 구조화 출력 → 파싱/재시도 → 평가)가 겹친다는 뜻. 입력 모달리티(텍스트 vs 이미지)만 다름.
- CLIP/임베딩 스코어링 쪽은 이런 "요리 비유"가 통하지 않음 — 재료를 접시에 담는 게 아니라 좌표 공간에 뿌리고 가까운 것을 찾는 완전히 다른 방식의 작업이므로 별도로 학습이 필요한 영역.
